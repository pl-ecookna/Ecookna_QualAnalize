# Ecookna QualAnalize (Система автоматической проверки заказов)

Проект представляет собой автоматизированную систему проверки заказов (в формате PDF) на соответствие техническим правилам, нормам "слипаемости" стеклопакетов и стандартам оформления.

## 📌 О проекте

Система принимает PDF-файлы заказов через Telegram-бота, извлекает сырой текст с помощью n8n, парсит его через JavaScript (Regex) и сохраняет структурированные данные в PostgreSQL, где выполняются строгие детерминированные проверки (SQL-процедуры). По результатам анализа формируется подробный отчет с указанием ошибок и предупреждений.

**Ключевая особенность:** Гибридная архитектура.
- **n8n + JavaScript**: Используется для извлечения данных (OCR + Regex parsing) из PDF.
- **Логика проверки (Validation Logic)**: Реализована на стороне PostgreSQL в виде хранимых процедур, что исключает ошибки при математических расчетах.

---

## 🛠 Технический стек

1.  **n8n**: Оркестрация процессов.
    -   Прием файлов от Telegram API.
    -   Маршрутизация и вызов внешних сервисов (DB).
    -   Отправка ответов пользователю.
2.  **PostgreSQL**: Основное хранилище и движок бизнес-логики.
    -   Хранение истории проверок (`qual_analize_files`).
    -   Хранение разобранных позиций заказа (`qual_analize_pos`).
    -   **Сложная валидация**: SQL-процедуры для проверки формул, размеров и комплектации.
3.  **JavaScript**: Парсинг текста с помощью регулярных выражений (Regex).
4.  **Telegram Bot API**: Интерфейс взаимодействия с пользователем.

## 📂 Структура проекта

### `/scripts` — База данных (PostgreSQL)

Ядро системы. Включает схемы таблиц и хранимые процедуры, реализующие бизнес-логику валидации.

#### 1. Таблицы (`/scripts/tables`)

| Таблица | Описание |
| :--- | :--- |
| **`qual_analize_files`** | Журнал обработанных файлов. Хранит метаданные (имя файла, chat_id пользователя), сырой ответ парсера и примененные правила. |
| **`qual_analize_pos`** | Детализированные данные по каждой позиции заказа. Содержит: размеры (сырые/округленные), формулу, артикулы (JSON), статус проверки (`overall_status`) и ссылки на таблицу слипания (`f1`, `f2`). |
| **`qual_analize_pos_issues`** | Найденные ошибки и предупреждения. Привязаны к позиции (`pos_id`). Поля: `issue_code`, `severity` (error/warning), `message`. |
| **`art_rules`** | Справочник артикулов. Содержит свойства стекол (тип, обработка, покрытие) для проверки соответствия. |
| **`qual_analize_rules`** | Справочник правил валидации (активность правил). |
| **`size_control`** | Таблица "слипаемости" (Size Control). Определяет допустимые формулы (`formula_1`, `formula_2`) для диапазонов размеров стеклопакетов. |
| **`qual_analize_prompts`** | Справочник промптов (используется для версионирования логики парсинга или legacy-функционала). |

#### 2. Хранимые процедуры и Триггеры (`/scripts/procedures`)

**Основные проверки:**

*   **`trg_calc_qual_pos`** *(Trigger)*: Срабатывает перед вставкой позиции.
    *   Округляет размеры до 50/100 мм.
    *   Считает количество камер.
    *   Находит соответствующую запись в `size_control` и заполняет поля `f1`/`f2` (допустимые формулы).
*   **`trg_qual_checks_after`** *(Trigger)*: Срабатывает после вставки. Запускает цепочку проверок.

**Функции валидации:**

*   **`check_slip(pos_id)`**: Основная проверка на слипание.
    *   Парсит фактическую формулу стеклопакета.
    *   Сравнивает толщины стекол и рамок с требованиями из `f1`/`f2`.
    *   Генерирует ошибку `SLIP_MISMATCH` с детализацией (где именно несоответствие).
*   **`check_argon(pos_id)`**: Проверяет заполнение газом. Если Аргон есть хотя бы в одной камере, он должен быть во всех.
*   **`check_slip_tempered(pos_id)`**: Проверяет требование закалки. Если таблица слипания требует закаленное стекло ("з"), а в заказе обычное — создает ошибку.
*   **`check_missing_glass(pos_id)`**: Сверяет артикулы стекол из формулы со справочником.
*   **`recalc_overall(pos_id)`**: Обновляет итоговый статус позиции (OK/WARN/ERROR) на основе найденных проблем.

**Вспомогательные функции:**

*   **`collect_file_issues_message(file_id)`**: Генератор отчета. Собирает все ошибки по всем позициям файла в один форматированный текст для отправки в Telegram.
*   **`parse_order_elements(formula)`**: Разбирает строку формулы (напр. `4M1-16Ar-4M1`) на элементы (стекла/рамки).
*   **`split_marking_formula_simple`**: Триггер для таблицы `size_control`, парсит строковые обозначения правил слипания.

### `/n8n_workflows` — Сценарии автоматизации
JSON-файлы рабочих процессов для импорта в n8n.

-   `1_QualGetFiles.json`: Сценарий получения файла, первичной обработки и отправки на распознавание.
-   `3_QualGenerateReport-2.json`: Сценарий генерации отчета и отправки результата в Telegram.

## 🚀 Как это работает (Data Flow)

1.  **Загрузка**: Пользователь отправляет PDF в Telegram-бот.
2.  **Триггер (n8n)**: Перехватывает файл, сохраняет метаданные в БД (`qual_analize_files`).
3.  **Распознавание и Парсинг**:
    -   n8n извлекает сырой текст из PDF.
    -   JavaScript-код (Regex) разбирает текст, выделяя параметры заказа (размеры, формулы, состав пакета).
4.  **Сохранение (DB)**: Структурированные данные записываются в таблицу `qual_analize_pos`.
5.  **Валидация (SQL Triggers)**:
    -   При вставке данных срабатывают триггеры (`trg_calc_qual_pos`, `trg_qual_checks_after`).
    -   Запускаются процедуры проверок (`check_slip`, `check_argon` и др.).
    -   Результаты проверок записываются в `qual_analize_pos_issues`.
6.  **Генерация отчета**:
    -   Вызывается функция `collect_file_issues_message(file_id)`.
    -   Она собирает все ошибки, форматирует их (добавляет иконки ⛔️/⚠️/ℹ️) и создает читаемый текст.
7.  **Ответ**: n8n забирает готовый текст и отправляет его пользователю в Telegram.

## ⚠️ Важные замечания

-   **Актуальность**: Текущая реализация (SQL-скрипты) является "источником правды". Документация в папке `docs` частично устарела (например, описание логики проверок через LLM).
-   **Зависимости**: Для работы требуются развернутый PostgreSQL и инстанс n8n.
